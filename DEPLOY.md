# Deploying to Render

This app needs a **long-running container** (it holds open live SSE streams and
runs debates in background threads with in-memory session state). Render's web
service fits that; serverless platforms like Vercel do not.

- This deployment runs in **managed-keys mode** (`MANAGED_KEYS=1` in
  `render.yaml`): users never enter API keys â€” the app always uses **your**
  `ANTHROPIC_API_KEY` / `OPENAI_API_KEY`. That means **every logged-in user spends
  your money.** Protect the app accordingly:
  - Set a strong `APP_PASSWORD` (or `APP_USERS`) and only share it with people you
    trust. Without login credentials the app is **open to anyone**.
  - Watch your Anthropic/OpenAI usage dashboards; consider setting spend limits
    there.
- In managed mode both API keys are **required** â€” without them, starting a debate
  returns a clear "missing keys" error.
- Always set a random `SECRET_KEY` (the blueprint auto-generates one).

> To instead let each visitor bring their own key (no owner cost), set
> `MANAGED_KEYS=0` and leave the API keys unset; the key fields reappear in the UI.

## One-time: put the code in a Git repo

Render deploys from Git. From the project directory:

```bash
git init && git add -A && git commit -m "AI vs AI debate app"
# create a repo on GitHub, then:
git remote add origin git@github.com:<you>/ai-vs-ai.git
git push -u origin main
```

`.env`, `.venv/`, and `debates.db` are already git-ignored, so no secrets ship.

### Free plan instead of paid

`render.yaml` mounts a 1 GB disk so the SQLite **history persists**. Disks need a
paid instance. To use the **free** plan: delete the `disk:` block **and** the
`DEBATE_DB` env var from `render.yaml`. The app still works, but history resets
on every restart/deploy.

## Manual setup (no blueprint)

Create a **Web Service** from the repo with:

- **Build command:** `pip install -r requirements.txt`
- **Start command:** `gunicorn app:app --workers 1 --threads 16 --timeout 0 --bind 0.0.0.0:$PORT`
- **Health check path:** `/healthz`
- **Env vars:** `APP_PASSWORD`, `SECRET_KEY`, `COOKIE_SECURE=1`, and optionally the
  API keys + `DEBATE_DB`.

## Why a single worker?

Sessions live in process memory and are shared between `/api/start`,
`/api/stream`, and follow-up requests. Run **one worker** (threads give you
concurrency). To scale horizontally you'd move sessions to Redis first â€” out of
scope for this simple build.

## Config reference

| Variable            | Purpose                                             |
|---------------------|-----------------------------------------------------|
| `APP_PASSWORD`      | Single shared login password.                       |
| `APP_USERS`         | `user:pass` pairs, comma-separated (alternative).   |
| `SECRET_KEY`        | Signs the login cookie. Random per-restart if unset.|
| `COOKIE_SECURE`     | `1` to send the cookie only over HTTPS (production).|
| `MANAGED_KEYS`      | `1` = always use the host's keys, hide the UI key fields (public mode). `0` = users bring their own. |
| `ANTHROPIC_API_KEY` | Claude key. Required when `MANAGED_KEYS=1`.         |
| `OPENAI_API_KEY`    | ChatGPT key. Required when `MANAGED_KEYS=1`.        |
| `DEBATE_DB`         | Path to the SQLite history file.                    |
| `PORT` / `HOST`     | Bind address (Render sets `PORT`).                  |
